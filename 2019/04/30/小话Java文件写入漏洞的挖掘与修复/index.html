小话Java文件写入漏洞的挖掘与修复

0x00 前言

“文件写入漏洞”是在Java Web应用中比较常见的一种漏洞。该漏洞的PoC如下：

这种漏洞的原因在于：部分程序在写入文件时，读取文件的参数filename直接在请求里传递，后台程序获取到这个文件路径后直接读取返回。问题在于这个参数是用户可控的，攻击者直接传入想要读取的文件路径即可利用。

小编前段时间在ZrLog 2.1.0这套Java博客系统的后台发现了一个文件写入漏洞，之后也在ZrLog的GitHub主页提了一个issue。不久后，官方推出的2.1.1版修复了这一漏洞。小编在这个过程中学习了一些东西，于是记了这篇笔记，请帮我指正！

这边博客将包含以下内容：

1）漏洞描述
2）漏洞出现的位置
3）漏洞复现
4）漏洞修复
5）反思

0x01 漏洞描述

ZrLog 是使用 Java 开发的博客程序。
ZrLog 2.1.0产品后台存在文件写入漏洞，攻击者可以利用产品后台的“设置/错误页面编辑”功能进行文件写入。通过利用该漏洞，攻击者可以进行“文件的跨目录写入”，甚至替换ZrLog网站的配置文件web.xml，致使网站崩溃。

该漏洞出现的文件路径为：\zrlog\web\src\main\java\com\zrlog\web\controller\admin\api\TemplateController.java，具体位置如下图所示。

这段代码是网站管理员在网站后台的“设置/错误页面编辑/提交”处进行错误页面自定义的部分代码。可以发现file这个字符串由PathKit.getWebRootPath()和getPara(“file”)这两个字符串拼接而成。其中PathKit.getWebRootPath()的返回值是网站根目录的路径。跟进getPara(“file”)方法，如下图所示。

可以发现该方法的返回值是this.request.getParameter(name)，而this.request是javax.servlet.http.HttpServletRequest对象，如下图所示。

由此可见，这套CMS在做“错误页面编辑”时，未进行任何过滤就进行了文件路径和文件名的拼接。

0x05 小总结

  1. 在挖掘“文件写入漏洞”的时候，可以考虑采用“黑白盒测试相结合”的方式（先在可能触发文件写入漏洞的点，借用BurpSuite等工具进行测试。接着，进行代码审计）；
  2. 为了解某开源程序是否存在漏洞，可以查看其GitHub等平台上的“issue”或者“commit”(有些有经验的漏洞挖掘者可能通过官方更新逆推出旧版本的漏洞)；
